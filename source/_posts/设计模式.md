# 面向对象设计原则

## 为什么

变化是复用的天敌，而面向对象设计模式的最大优势是抵御变化。

从底层思维来说， **继承**、**封装**、**多态**。

1. 从宏观上来讲，面向对象设计就是隔离变化，封装变化点。
2. 从微观层面来说，面向对象是各司其责，强调每个类的责任。也就是由于需求变化导致的新增类型不应该影响原来类型的实现。

## 慎用设计模式

> 如果你有一把锤子，那么你看什么东西都像钉子。

初学设计模式的时候很容易走入一个误区：在学习了某个设计模式之后会在任何地方都是用这个模式，即使是更简单的代码完全能胜任的时候。

有时候我们将，好的代码并不是一蹴而就的，好的代码应该是重构出来的。

## 设计原则

在GOF《设计模式：可复用面向对象软件的基础》这本书中，提到了23种设计模式，但是有些设计模式可能被一些语言机制所替代，举个简单的例子，策略模式在现代语言中（包括C++）都可以使用lambda表达式实现。有些设计模式可能不再流行，但是只要能够遵循设计原则甚至可以发明自己的设计模式。
学习设计模式不能像学习算法一样，比如对于C++来说，如果了解虚函数的调用机制，23种设计模式可以非常快的看明白，但是在实际使用的时候可能就会因为某个点的改动而跟原先的设计模式完全不同。所以有时候看起来两段非常相似的代码，其实是完全不同的两个模式，而有时候两段代码天差地别，但是他俩是同一个设计模式。衡量的标准就是设计原则。


1. 依赖倒转原则
    上层模块不应该依赖下层模块，下层模块也不应该依赖上层模块，二者都应该依赖抽象。抽象不应该依赖实现细节，实现细节应该依赖抽象。依赖倒转原则主要描述了针对接口编程，如果上层模块依赖了下层模块，则当下层模块不稳定的时候上层模块也将会不稳定，如果抽象依赖了实现，通常实现会是不稳定的，则抽象也会变得不稳定，但是抽象出来的东西应该一定是稳定的。因此上下层模块不应该相互依赖，而二者应该去依赖一个稳定的东西（抽象），上下层模块都可以理解成实现，则实现应该依赖抽象而不是用抽象去依赖实现。
2. 开闭原则
    对扩展开放，对修改封闭。类模块应该是可扩展的，而不是可修改的。
3. 单一职责原则
    一个类应该只有一个引起他变化的原因，而变化的方向隐含着类的责任。
4. Liskov替换原则
    子类必须能够替换他们的基类（is-a）。
5. 接口隔离原则
    不应该强迫用户程序依赖他们不用的方法，接口应该小而完备。子类使用：protected，本类使用：private。如果过多的暴露接口，客户程序就会对本类产生依赖，那么接口就需要保持问题。
6. 优先使用对象组合而不是类继承
    继承破坏了封装性，父类给子类暴露了过多的接口，但是组合只要有良好的接口设计耦合度低。
7. 封装变化点
    使用封装创建对象之间的分界层，设计者可以在分界点的变化的一侧进行修改而保持稳定的部分不受影响。
8. 针对接口编程而不是针对实现编程。
    客户程序只需要知道对象的接口而不需要知道对象的具体类型，实现高内聚和松耦合。

# 模板方法

在我以前的开发经历中会想当然的这样做：在我编写的软件中去写流程，并且去调用三方库，就好像我的项目中需要五个步骤，其中135步骤的功能都是由三方库提供的，我在自己的软件中负责编写24步骤。

```c++
class lib {
public:
    step1();
    step3();
    step5();
};
class app {
public:
    step2();
    step4();
};
int main() {
    lib::step1();
    if(app::step2()) {
        lib::step3();
    }
    if(app::step4()) {
        lib::app5();
    }
}
```

上面的流程中lib类是库作者提供给我们的，而app类和main函数中的流程则是我们自己编写的，设计模式要做的就是找到稳定和变化之间的分割点，也就是通过设计模式将稳定和变化进行隔离，而在这里如果整个流程是确定的（稳定）那么如果我们每次都需要编写整个流程的话这就模糊了变化和稳定的分界点。
如果使用模板方法对上述进行改进的话可以做出一下流程。

```c++
class lib {
public:
    run() {
        step1();
        if(step2()) {
            step3();
        }
        if(step4()) {
            app5();
        }
    }
protected:
    virtual step2() {}
    virtual step4() {}
private:
    step1();
    step3();
    step5();
};

class app : public lib {
    step2() override {}
    step4() override {}
};

int main() {
    app::run();
}
```

对比一下上面使用模板方法重新设计过的流程，差别就在于将软件的流程（稳定）挪到了库中进行实现，也就是实现了一个晚绑定

> 早绑定与晚绑定
> 
> 当在我们的项目中需要引用三方库的时候如果不使用任何模式的话，最直观的思考方式就是就是用我们的代码去调用库代码，换句话说我们的调用顺序是跟我们代码的存在顺序是一直的，也就是先有三方库后我们代码（后出现）通过函数调用或者其他方式调用原先就存在的代码，按理说先有三方库，后我们通过调用三方库实现我们的代码流程。
>
> 而晚绑定跟代码的存在顺序是不一致的，也就是在三方库中流程就已经存在了，但是这个流程中有一些步骤是需要我们应用编写者进行实现的，也即是我们需要将某些步骤实现之后整个流程才算完整，这就是晚绑定。

使用了模板方法之后我们只要在类中继承库而后重写库函数中的虚函数即可，这里的虚函数可以是有定义的普通虚函数，也可以是抽象类的纯虚函数，这里是根据实际的代码情况来指定的（再次重申：设计模式重要的是思想而不是实际的代码）

# 工厂方法

工厂方法是将对象的创建从基类推迟到子类，一种使用模板方法实现工厂方法的方式如下：

**Shape.h**

Shape基类定义一个具体形状的接口，业务代码使用这个基类来实现运行时多态。在这个命名空间中创建了一个辅助函数用来创建一个具体的对象，这个函数通过返回一个基类的智能指针让业务代码实现运行时多态。

```C++
#ifndef SHAPE_H
#define SHAPE_H

#include <iostream>
#include <memory>
namespace shape {

class Shape {
public:
    virtual void draw() = 0;
    virtual ~Shape() = default;
};

template <typename T>
std::unique_ptr<Shape> createShape() {
    return std::make_unique<T>();
}
};  // namespace shape
#endif  // SHAPE_H

```

具体的形状只需要继承基类，并且重写基类中的代码即可实现。

**Circle.h**

```C++
#ifndef CIRCLE_H
#define CIRCLE_H

#include "Shape.h"

namespace shape {

class Circle : public Shape {
public:
    void draw() override {
        std::cout << "Drawing Circle." << std::endl;
    }
};

}  // namespace shape
#endif  // CIRCLE_H

```

**Rectangle.h**

```C++
#ifndef RECTANGLE_H
#define RECTANGLE_H

#include "Shape.h"

namespace shape {
class Rectangle : public Shape {
public:
    void draw() override {
        std::cout << "Drawing Rectangle." << std::endl;
    }
};
}  // namespace shape

#endif  // RECTANGLE_H

```

**main.cc**

在实际的业务代码中，通过模板函数将某个类型传入来创建一个对象，使用auto来进行自动推导类型，这个类型应该是`std::unique_ptr<Shape>`类型，之后就可以展现运行时多态。

```C++
#include <memory>

#include "Circle.h"
#include "Rectangle.h"
#include "Shape.h"

int main() {
    auto circle = shape::createShape<shape::Circle>();
    circle->draw();
    auto rectangle = shape::createShape<shape::Rectangle>();
    rectangle->draw();
}
```

但是这么写也有一定的问题，主要是扩展性的问题，如果需要创建的对象是从配置文件中读取的，那么使用模板方法会受到限制。也就是在编译之前我们就已经确定好那些对象会被创建哪些不会被创建。

使用反射是一种可能的解决方法。
